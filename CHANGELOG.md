- **Conflicts externalized into a synced referential; tags reduced to `capture_source` + free** ([`backend/app/models/conflict.py`](backend/app/models/conflict.py), [`backend/app/routers/conflicts.py`](backend/app/routers/conflicts.py), [`backend/app/schemas/conflict.py`](backend/app/schemas/conflict.py), [`backend/app/services/conflict_sync.py`](backend/app/services/conflict_sync.py), [`backend/scripts/sync_conflicts.py`](backend/scripts/sync_conflicts.py), [`backend/scripts/seed_conflicts.py`](backend/scripts/seed_conflicts.py), [`backend/alembic/versions/j2l4n6p8r0t2_conflicts_referential.py`](backend/alembic/versions/j2l4n6p8r0t2_conflicts_referential.py), [`backend/app/services/events.py`](backend/app/services/events.py), [`backend/app/routers/events/`](backend/app/routers/events), [`docs/data-model.md`](docs/data-model.md), [`docs/api.md`](docs/api.md), [`docs/ingestion.md`](docs/ingestion.md), [`planning/next.md`](planning/next.md)). Conflicts stop being `tags` rows (`category='conflict'`) and become a dedicated **`conflicts` referential** (`name`, nullable-unique `wikidata_id`, `start_year` / `end_year`, `ongoing`, `tier`, `last_seen_at`, `source ∈ sync|seed|manual`) plus an `event_conflicts` M2M; migration `j2l4n6p8r0t2` moves the existing conflict tags and their event links over (`source='manual'`), deletes the conflict tag rows, and seeds the `Other` escape value. The list is **externally synced**: [`services/conflict_sync.py`](backend/app/services/conflict_sync.py) parses Wikipedia's "List of ongoing armed conflicts" via the MediaWiki API (three top tiers, skirmishes excluded) and upserts by **Wikidata QID, not name**, because the page renames conflicts constantly (24 of 35 month transitions over 36 monthly snapshots, 2023-2026, changed at least one name; Sudan carried 5 names in 3 years): a rename updates `name` in place, a QID-less same-name row is adopted, a collision is skipped and logged. Each pass also records the row's Wikipedia death-toll **`tier`** (`major` 10,000+ combat deaths in the current or previous year, `minor` 1,000-9,999, `conflict` 100-999), taken from which tier table the row sits in and updated when a conflict moves tiers (NULL for rows the sync has never seen), and parses the row's start-of-conflict year to backfill `start_year` where it is NULL (the seed's years are never overwritten). Disappearance from the page flips `ongoing=false` only after a 14-day grace period (`last_seen_at`); rows are **never deleted**, so ended conflicts stay selectable for archival footage, and rows the sync has never seen are immune. Parsing is strict: structural mismatch or a row count outside [15, 80] aborts with no write, and the runner (`scripts/sync_conflicts.py`, meant for a daily Railway cron) exits non-zero. A one-shot Wikidata SPARQL seed (`scripts/seed_conflicts.py [--dry-run]`) provides the historical depth: ~700-850 conflicts since 1914 through a P31 type allowlist (wars, civil wars, rebellions, insurgencies…; battles / operations / coup attempts excluded), inserted as `source='seed'`, `ongoing=false`, never modifying existing rows. On the API: a public **`GET /conflicts`** (60/min, ongoing first then by name, `tier` exposed on `ConflictRead`, `?used=true` narrowing to conflicts on at least one live event, mirroring the `GET /tags` orphan filter); event writes (`POST /events`, `POST /events/requests`, `POST /events/{id}/geolocate`) take a `conflict_ids` multipart field and the evidence floor becomes one conflict + one `capture_source` tag ("A conflict is required", still `tag_requirements_not_met`); the `conflict` filter on `GET /events` / `/events/points` now matches `conflicts.name`; `EventRead` / `EventList` gain `conflicts`; `GET /tags?curated=true` returns `capture_source` only. Product-wise the submit picker's default list shows the major-tier conflicts with the `Other` escape last, search covers the whole referential, an "include ended conflicts" toggle brings in ended ones, every conflict displays its years (`start_year`/`end_year`, "present" when ongoing) unless its name already embeds them, and the map filter shows only used conflicts. Wikipedia content is CC BY-SA; the attribution note lives next to `GET /conflicts` in [`docs/api.md`](docs/api.md).

- **Session-lifecycle invalidation via `users.token_version`** ([`backend/alembic/versions/t6o8q0s2u4w6_user_token_version.py`](backend/alembic/versions/t6o8q0s2u4w6_user_token_version.py), [`backend/app/models/user.py`](backend/app/models/user.py), [`backend/app/services/auth.py`](backend/app/services/auth.py), [`backend/app/dependencies.py`](backend/app/dependencies.py), [`backend/app/routers/auth.py`](backend/app/routers/auth.py), [`backend/app/services/admin.py`](backend/app/services/admin.py), [`backend/tests/test_session_invalidation.py`](backend/tests/test_session_invalidation.py)). Until now, clearing the session cookie did not invalidate the underlying JWT; a leaked cookie (XSS sniff, stolen device, ex-tenant of a shared laptop) stayed live to its `exp`. New `users.token_version INTEGER NOT NULL DEFAULT 0` column is embedded as a `tv` claim by `create_access_token` and re-checked in `get_current_user`; mismatch returns the same opaque 401 as every other decode failure. A new `bump_token_version(user)` helper fires at the four mutation points: `/auth/logout`, `/auth/change-password`, `/auth/reset-password`, and admin `soft_delete_user`, so each one invalidates every outstanding session for the affected user. `change-password` re-issues a fresh cookie on the same response so the device the user is on stays logged in while their other sessions get logged out. `/auth/reset-password` additionally gains mint-side parity guards: it now rejects soft-deleted and deactivated users (matching `/login` and the forgot-password mint), so a reset token captured before an account was disabled can no longer rotate it back into a usable credential. The migration backfills `0` on existing rows (PostgreSQL metadata-only fast path); pre-migration cookies (no `tv`) 401 on the first post-deploy request, the intended one-time forced logout. The full Tier-3 refresh-token system planned for the open-beta milestone supersedes this; this is the interim Tier-2 fix that closes the cookie-clear-isn't-invalidation gap now. Nine lock-in tests in `test_session_invalidation.py` cover the four bump sites + mint-parity + the tampered-cookie-doesn't-bump-arbitrary-users guard.

- **DCO sign-off enforcement on inbound contributions** ([`CONTRIBUTING.md`](CONTRIBUTING.md), [`docs/engineering.md`](docs/engineering.md)). Every commit on a PR must carry a `Signed-off-by:` trailer; the first one without it fails the PR. - **CI workflows hardened: every action SHA-pinned, least-privilege `permissions:` blocks** ([`.github/workflows/backend.yml`](.github/workflows/backend.yml), [`.github/workflows/frontend.yml`](.github/workflows/frontend.yml), [`.github/workflows/deploy.yml`](.github/workflows/deploy.yml), [`.github/workflows/doc-sync.yml`](.github/workflows/doc-sync.yml), [`.github/workflows/pr-title.yml`](.github/workflows/pr-title.yml)). Floating `@v4` / `@v5` major-version tags replaced with full commit SHAs (`actions/checkout` v4.3.1, `actions/setup-node` v4.4.0, `astral-sh/setup-uv` v3.2.4, `amannn/action-semantic-pull-request` v5.5.3) so a tag-republish from a compromised maintainer can't silently swap our build steps. Every workflow now declares a top-level `permissions:` block scoped to the minimum it actually needs (`contents: read` for the four CI workflows; `pull-requests: read` already on `pr-title.yml`); the workflow `GITHUB_TOKEN` no longer ships the default-broad scope. No trigger or job-logic changes; this is hardening, not refactoring. Confirmed no workflow uses `pull_request_target`. Sequenced for the open-source flip; public forks make every workflow run an attacker-reachable surface, so the lockdown lands before, not after. (#102)
- **Per-IP rate limits now actually per-IP, and immune to `X-Forwarded-For` spoofing** ([`backend/Dockerfile`](backend/Dockerfile), [`backend/app/services/audit.py`](backend/app/services/audit.py), [`backend/app/main.py`](backend/app/main.py), [`backend/app/routers/auth.py`](backend/app/routers/auth.py), [`backend/app/routers/admin.py`](backend/app/routers/admin.py), [`backend/app/routers/bounties.py`](backend/app/routers/bounties.py), [`backend/app/routers/geolocations.py`](backend/app/routers/geolocations.py), [`backend/app/routers/search.py`](backend/app/routers/search.py), [`backend/tests/test_auth_events.py`](backend/tests/test_auth_events.py)). Two coupled changes: (1) the Dockerfile now passes `--proxy-headers --forwarded-allow-ips='*'` so uvicorn honours `X-Forwarded-For`; without this every per-IP rate limit (`/login` 5/min, `/register` 10/hr, `/forgot-password` 5/hr, `/reset-password` 10/hr, the global 60/min) collapsed to one bucket keyed on the Railway-edge IP. (2) Every `slowapi.Limiter` now uses a new shared `rate_limit_key` that calls the existing right-most-XFF parser `extract_client_ip` (mirroring the audit-log path). The cross-check on this one matters: the obvious fix (`get_remote_address` + `--forwarded-allow-ips='*'`) would have been *strictly worse* than the original bug; uvicorn's `always_trust=True` branch returns `x_forwarded_for_hosts[0]` (the LEFT-most XFF entry), and Railway appends rather than overwrites XFF, so a left-most-keyed limiter is attacker-controlled (rotate `X-Forwarded-For: <random>` for a fresh bucket per request, or send `X-Forwarded-For: <victim_ip>` to lock a victim out). Routing slowapi through the same right-most parser as the audit log closes both vectors. Two new lock-in tests in `test_auth_events.py` assert the spoof can't mint a fresh bucket and can't pin a victim's.
- **Tiptap proof images now reject protocol-relative URLs** ([`backend/app/services/sanitize.py`](backend/app/services/sanitize.py), [`backend/tests/test_sanitize.py`](backend/tests/test_sanitize.py)). `_safe_image_src` returned early on `value.startswith("/")`, which matched `//attacker.example/pixel.gif`. The browser resolves protocol-relative URLs against the page scheme, so a persisted tracking pixel leaked every viewer's IP/UA/Referer, defeating the central anti-tracking-pixel claim of the sanitiser. Now rejected with the rest of the unsafe-host paths.
- **Admin-escalation via email case fixed** ([`backend/app/schemas/__init__.py`](backend/app/schemas/__init__.py), [`backend/app/schemas/auth.py`](backend/app/schemas/auth.py), [`backend/app/schemas/recovery.py`](backend/app/schemas/recovery.py), [`backend/tests/test_registration_pending.py`](backend/tests/test_registration_pending.py)). `User.email` is stored case-preserving with a case-sensitive UNIQUE, but `maybe_promote_admin` compares `user.email.lower()` against `ADMIN_EMAILS`. An attacker with an invite code could register `Admin@vidit.app` alongside the canonical `admin@vidit.app` and auto-promote to `is_admin=True` on first login. Fixed by normalizing every incoming email to lowercase at the schema layer (new shared `NormalizedEmail = Annotated[EmailStr, AfterValidator(_lowercase)]`), so the UNIQUE catches the case-variant on registration and the login lookup finds the right row regardless of how the user typed their address.
- **HTTP-layer request-body size cap** ([`backend/app/main.py`](backend/app/main.py), [`backend/app/config.py`](backend/app/config.py), [`backend/tests/test_health.py`](backend/tests/test_health.py)). Per-file size limits in `storage.validate_file` ran AFTER Starlette had already buffered the entire multipart body to a `SpooledTemporaryFile`: advisory, not protective. New `enforce_request_body_size` middleware rejects a request with `Content-Length` above `max(max_video_size, max_files_per_geolocation × max_image_size) + 10 MB` headroom (~130 MB today) with a clean 413 before any bytes touch the worker. Also rejects negative `Content-Length` (which would otherwise parse cleanly via `int()` and slip past a pure `>` check). CORS sits *outside* the middleware so cross-origin browsers see the 413 with `Access-Control-Allow-Origin` stamped instead of a "CORS error" in DevTools. The cap moved through three iterations during PR #100 review: an initial `max_video_size + 10 MB` (= 110 MB) silently 413'd a legitimate full 12-image batch, then the cap was widened to the max-of formula and the middleware re-ordered inside CORS, then the negative-CL guard was added. Three lock-in tests in `test_health.py`.
- **Per-submission file count cap on `POST /geolocations`** ([`backend/app/config.py`](backend/app/config.py), [`backend/app/routers/geolocations.py`](backend/app/routers/geolocations.py), [`backend/tests/test_geolocations.py`](backend/tests/test_geolocations.py)). New `max_files_per_geolocation = 12` setting (lives in `config.py` so the body-size middleware reads it without forming a `main → routers` import edge) rejects multi-thousand-file payloads that would otherwise pin the worker through Pillow + EXIF strip + JPEG re-encode + 2 derivatives + 3 S3 PUTs per file. Returns 422 to match the rest of this router's malformed-input rejections (`event_date`, `_parse_bbox`, `_parse_filter_date`).
- **S3 keys derive their extension from the validated MIME, not the filename** ([`backend/app/services/storage.py`](backend/app/services/storage.py), [`backend/tests/test_storage_local.py`](backend/tests/test_storage_local.py)). `upload_file` / `upload_bounty_file` / `upload_proof_image` previously concatenated `Path(file.filename or "").suffix.lower()` into the S3 key. Attacker-controlled filenames could push the key past S3's 1024-byte limit, smuggle RTL-overrides, or land an `evil.html` suffix on a `video/mp4` payload. New `_safe_storage_extension` maps the (already-validated) content type to a short ASCII suffix.
- **Date-filter params no longer 500 on garbage** ([`backend/app/routers/geolocations.py`](backend/app/routers/geolocations.py), [`backend/tests/test_geolocations.py`](backend/tests/test_geolocations.py)). `_apply_filters` was concatenating `submitted_to + " 23:59:59"` and forwarding the raw `event_date_*` strings; Postgres raised `InvalidDatetimeFormat` and surfaced as 500. `/points` is anonymous-reachable in M3, so unauthenticated scrapers could fill Sentry with 500s. New `_parse_filter_date` validates with `date.fromisoformat` and returns 422; the call truncates input to 10 chars first so a previously-working `2026-05-01T12:00:00` URL doesn't 422 on the tightened shape. `submitted_to` uses the cleaner `< date + timedelta(days=1)` (end-of-day inclusive without the string-concat trick).
- **`POST /geolocations` validates `event_date` / `title` / `source_url` before any S3 round-trip** ([`backend/app/routers/geolocations.py`](backend/app/routers/geolocations.py), [`backend/tests/test_geolocations.py`](backend/tests/test_geolocations.py)). `event_date` was a bare `Form(str)`; `'not-a-date'` flowed through to `Mapped[date]` and 500'd at flush AFTER files had landed in S3 (orphan-cleanup branch fired but burned the upload). `title` / `source_url` were unbounded `Form(str)` and overflowed the `String(255)` / `String(2000)` columns the same way. Now parsed (with `date.fromisoformat`) and length-checked at the boundary; malformed `event_date` returns 422 to match `_parse_filter_date` / `_parse_bbox`.
- **`POST /tags` and `POST /bounties/{id}/claim` no longer 500 on a concurrent-write race** ([`backend/app/routers/tags.py`](backend/app/routers/tags.py), [`backend/app/routers/bounties.py`](backend/app/routers/bounties.py)). Both endpoints did SELECT-then-INSERT without a SAVEPOINT, so two concurrent POSTs both passed the existence check and one lost the UNIQUE / composite-PK race with an unhandled `IntegrityError`. Now wrap the INSERT in `db.begin_nested()` and map the loser to the friendly outcome (`/tags` → 409 "Tag already exists", `/bounties/{id}/claim` → 204 idempotent as its docstring promises). Mirrors the pattern already in `services/social.follow_user`.

Second hotfix for v0.0.6's icon-bundling regression. v0.0.7 fixed the `/icon/*` 500s but missed that every dynamic page route bundles the icon module for metadata resolution and re-executes it at request time on the Vercel function.

### Fixed
- **Dynamic pages still 500'ing post-v0.0.7** with the same `ENOENT: Montserrat-700.ttf` from the icon-route font bundling fix. v0.0.7 traced the `.ttf` into the `/icon` and `/apple-icon` route bundles, which is what the curl-level 500 visibly proved. What v0.0.7 missed: Next.js's metadata-icon resolver bundles `app/icon.tsx` into **every page's** bundle too (`d.metadata.icon` in the compiled `page.js`), and dynamic pages (`/geolocations/[id]`, `/bounties/[id]`, `/profile/[username]`) re-execute it at request time on the Vercel function. Static pages were fine; they prerender metadata at build, so the .ttf was on disk in the build container. Broadened `experimental.outputFileTracingIncludes` to `"**"` so the font travels with every function bundle (~30 KB per function; acceptable, and the enumerate-each-dynamic-route alternative is brittle: a new `[id]` page would silently start 500'ing while local dev keeps passing because `process.cwd()` resolves the on-disk tree).

## v0.0.7, 2026-05-13

Hotfix for v0.0.6: the icon routes added in #60 were 500'ing in prod every request, cascading into a Server Components render error on every page that pulls the web app manifest.

### Fixed
- **Icon routes 500'ing in prod**, cascading into a Server Components render error on every page that pulls the manifest (which is every page). The generated icon / apple-icon routes load Montserrat-700 via `readFileSync(join(process.cwd(), "src/app/Montserrat-700.ttf"))`, a runtime-resolved path that Vercel's node-file-trace can't discover statically, so the .ttf wasn't bundled into the function and every `/icon/favicon`, `/icon/android-192`, `/icon/android-512`, and `/apple-icon` request landed on `ENOENT`. Fixed by adding `experimental.outputFileTracingIncludes` entries in `next.config.mjs` pointing both routes at the .ttf. The alternative `readFileSync(new URL("./Montserrat-700.ttf", import.meta.url))` shape works in plain Node but fails the Next.js build with `Received an instance of URL` because webpack rewrites `import.meta.url` for the server bundle. Slipped through v0.0.6 because the icon routes worked locally (`process.cwd()` resolves to the on-disk `frontend/` tree) and the Vercel preview deploys had been hitting the prod build for a different reason, first surfaced when a fresh prod deploy landed under the v0.0.6 tag.

## v0.0.6, 2026-05-13

The social-features release. Three Phase-2 product slices land together: bounties end-to-end, search slice 1 (Postgres FTS over geolocations, bounties, and analysts), and followers + a personal timeline. Editable analyst profiles ship alongside (bio, avatar, Linktree-style external links). The last auth-hardening tier closes (Tier 4 lite: HSTS + the `auth_events` audit log). A PyJWT-for-`python-jose` migration retires the unmaintained dependency, `?bbox=` malformed input now 422s instead of silently dropping the filter, and the bounty fulfilment path picks up race-safety + form-tamper-safety hardening from the post-merge review. Branded favicon, Apple touch icon, and Android web manifest land in the same window so the first analyst session no longer reads as "no icon anywhere".

### Changed
- Swapped `python-jose` → `PyJWT`. `python-jose` has been effectively unmaintained for years and has had CVEs; PyJWT is the actively-maintained equivalent in the same API shape (`jwt.encode` / `jwt.decode`). One-line import swap at three call-sites (`services/auth.py`, `dependencies.py`, `routers/auth.py::logout`); `JWTError` → `jwt.InvalidTokenError`. No behavior change other than PyJWT emitting an `InsecureKeyLengthWarning` when the HMAC secret is <32 bytes (prod uses `openssl rand -hex 32` per CLAUDE.md and is unaffected; the warning shows on the local dev secret).
- `GET /geolocations?bbox=...` now returns 422 on malformed input instead of silently dropping the filter and returning every row in the catalog. The parser validates four comma-separated floats, latitudes in [-90, 90], longitudes in [-180, 180], south ≤ north, west ≤ east. Empty result is the right fail-safe for a map endpoint; an unbounded result is not.
- **Bounty fulfilment is now race-safe and form-tamper-safe.** Post-review hardening of the bounty slice: (1) `POST /geolocations bounty_id=…` overrides the form's `title`, `source_url`, and `tag_ids` with the bounty's values server-side; the frontend renders these read-only when arriving via `?bounty_id=…`, but the API is the authority, so a hostile caller can't rewrite somebody else's bounty into an unrelated geolocation by lying about these fields. (2) The fulfilment path takes a `SELECT ... FOR UPDATE` on the bounty row and migration `n0i2d4e6f8a0` adds a partial unique index `WHERE originated_from_bounty_id IS NOT NULL`; concurrent fulfilment attempts on the same bounty serialise on the lock and the second one observes `status=fulfilled` + 409s; even if a future code path forgets the lock, the DB refuses the duplicate insert. (3) `DELETE /bounties/{id}` takes the same row lock so a concurrent fulfilment can't race it. (4) `Bounty.fulfilled_by` now filters `Geolocation.deleted_at IS NULL` in its primaryjoin so a soft-deleted geolocation no longer leaks back through the bounty detail. (5) `DELETE /admin/users/{id}` cascade-soft-deletes (or hard-deletes) the user's bounties alongside their geolocations; banning a user no longer leaves their open bounties orphaned on the queue or their fulfilled bounties surfacing their banned handle. The response now carries `cascaded_bounties` alongside `cascaded_geolocations`; the bounty seeder now spreads the demo set across the lifecycle (70% open / 15% fulfilled / 15% closed) and mints a paired demo geolocation for every fulfilled bounty so the status-filter chips, "Fulfilled by" link, and "originally posted as a bounty" trace banner all have data to render. Bounty `is_demo` is now serialised on the API and used to swap the synthetic source_url for a "synthetic" label so testers don't click out to a 404 (the always-attached `demo` tag remains the visible "this is demo data" signal, no dedicated pill).

New endpoints: `POST /users/{username}/follow` 🔒 + `DELETE /users/{username}/follow` 🔒 (both idempotent → 204), and `GET /timeline` 🔒 returning a `PaginatedGeolocations` feed of geolocations submitted by the analysts the caller follows, ordered by event date desc with `ST_X` / `ST_Y` pulled in the same SELECT (no per-row N+1). The unfollow endpoint returns 404 on an unknown username so a typo surfaces an error rather than silently no-op'ing. `GET /users/{username}` gains `followers_count`, `following_count`, and `is_following` (the last one is `true` only when the caller is authenticated and follows this user; anonymous and self-views always get `false`); the username probe accepts an optional auth cookie for that reason. - **Search: slice 1.** Postgres full-text search across the three first-class entity types (geolocations, bounties, analysts). Migration `o1j3k5l7m9n1` adds three GIN indexes over `to_tsvector('simple', …)` expressions: `geolocations(title)`, `bounties(title)`, `users(username || ' ' || bio)`. `simple` (not `english`) because the searchable corpus is heavy on analyst handles, place names, and conflict tags that don't stem cleanly; keeps "Donetsk" matching "Donetsk" exactly and not "donetsks". Control bytes rather than an ASCII marker so a user can't plant the sentinels in their own bio / title to corrupt parity for readers; the SQL also strips both bytes from the source document before `ts_headline` runs. Soft-deleted rows are filtered at query time on every branch. `bio_highlight` is intentionally `null` when only the username matched so the UI can hide the snippet block instead of rendering an un-highlighted bio. Per-group `total` is computed via a `COUNT(*) OVER ()` window function so it reflects the pre-LIMIT match count; the UI can render "3 of 142" truthfully instead of "3 of 3" tautologies. **Frontend.** Replaces the WipCard mockup at `/search` with a real implementation: debounced query input (300ms), URL-synced `?q=` + `?type=` so links share state, type-filter chips (All / Geolocations / Bounties / Analysts), three result groups stacked with per-card patterns reused from the existing index views, out-of-order response guard via a monotonic request counter (so quickly toggling chips with the same `q` can't land an older response on top of a newer one). Sidebar's Search entry loses its `wip: true` flag. **Slice 1 explicitly defers** filter chips beyond the type pick (date / conflict / tag / author / vetted-only, slice 2), JSONB-content search (`proof`, `bounty.description`), URL-fragment search, and per-group infinite scroll. Migration `m9h1c3d5e7f9` adds `bounties.is_demo BOOLEAN` + partial index `WHERE is_demo = true`. `DELETE /admin/seed-demo-bounties` 🛡️ drops every flagged row in one bulk DELETE; demo users and demo geolocations stay (separate panel), so an admin can keep one population while wiping the other. Migration `k7f9a1b3c5d7` creates `bounties` + `bounty_tags`, makes `Media` polymorphic via a nullable `bounty_id` with a `CHECK` constraint XOR'ing against `geolocation_id`, and adds `geolocations.originated_from_bounty_id` for the trace.   **Endpoints.** `POST /bounties` 🔒 (multipart, ≥1 media file, sanitised description). `GET /bounties` with `status` / `tag` / `author` filters; list response carries `claimer_count` + a denormalised `claimer_sample` (capped, newest first) so the index avoids N+1. `GET /bounties/{id}` with the full `claimers` array, `fulfilled_by`, and the rest of the read shape. `DELETE /bounties/{id}` 🔒 (author-only; 409 when a geolocation already traces back). `POST /bounties/{id}/claim` 🔒 (idempotent, rejected on non-`open` status). `DELETE /bounties/{id}/claim` 🔒 (no-op when caller wasn't a claimer). `POST /bounties/{id}/close` 🔒 (author-only withdraw without fulfilment, stamps `closed_at`).

The submission form reads the query param, fetches the bounty, prefills title / source_url / tags and renders the bounty's media as a read-only gallery; the analyst supplies coordinates, event date, and the proof body. The resulting geolocation detail page surfaces the bounty trace in an emerald banner linking back to the original.

  **Why this design.** Bounties without coordinates cannot appear on the map, which is intentional; they live exclusively on `/bounties` until promoted. The "claimed" lifecycle state from the initial sketch was dropped because single-claimer reservation overfits the use case: signaling presence is useful even when others are also signaling, so the lifecycle has three states (`open`, `fulfilled`, `closed`) and claims are an orthogonal multi-row signal. See `docs/next.md` → *Bounties* for the small list of possible follow-ups (notifications, claim TTL).

Two paths: (a) re-submitting `/register` with an email that already has a live pending row auto-routes to `/registration-pending?email=…` (where the Resend button lives) instead of showing a dead-end "in flight" 409; (b) new `/resend-confirmation` page linked from `/login` as "Didn't receive your confirmation email?": single email input, always returns the same UX regardless of whether the address matched, mirroring the `/forgot-password` anti-enumeration discipline. Standard pattern for hard-pre-creation verification flows (Mailchimp, Substack, AWS).
- Frontend auth URLs flattened: `/auth/login` → `/login`, etc. The five auth pages (`/login`, `/register`, `/forgot-password`, `/reset-password`, `/verify-email`) now live in a `(auth)` route group; the closed-beta gate moved to `(gate)/welcome/`. Backend API endpoints (`/api/v1/auth/*`) are unchanged; that's normal API hygiene, not user-visible. The split makes "delete the closed beta on public-launch day" a clean `rm -rf app/(gate)/ + lib/gate*.ts + app/api/gate/`. ([#41](https://github.com/vidithq/vidit/pull/41))

### Security
- Cut third-party asset fetches from authenticated pages. `/timeline`, `/search`, `/profile` mocks render via the new deterministic `MediaPlaceholder` (no network). Orphan `LocationPicker` + `leaflet` deps deleted. ([#40](https://github.com/vidithq/vidit/pull/40))
- Atomic `auth_tokens.consume()`: the recovery-token redemption uses a single `UPDATE … WHERE consumed_at IS NULL … RETURNING` so two concurrent requests for the same token can't both win. Closes a TOCTOU race that would have let an attacker holding a stolen reset link race the legitimate user. ([#41](https://github.com/vidithq/vidit/pull/41))
- Removed the unguarded `POST /auth/invite-codes` endpoint: any logged-in user could mint codes. Superseded by `/admin/invite-codes`, behind `require_admin`. ([#45](https://github.com/vidithq/vidit/pull/45))
- Closed three follow-ups from the admin-page review. `search_users` now filters `deleted_at IS NULL` and `set_user_trust` 404s on tombstoned rows so soft-deleted accounts can't surface in admin search or have their trust flag flipped. `/auth/login` always pays one bcrypt; the unknown-email and soft-deleted-user branches now verify against a pre-computed dummy hash rather than short-circuiting, closing a timing oracle that distinguished "no such email" from "wrong password against a live user". `/auth/forgot-password` had the same class of bug; the live-user branch ran a DB update, token mint, and synchronous Resend round-trip while the no-user branch returned immediately. The token mint + email send now run in a FastAPI `BackgroundTasks` worker dispatched after the 204, so both branches return at the same point. The endpoint's docstring previously claimed wire timing was identical; now it actually is.
