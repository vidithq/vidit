## v0.4: Curated onboarding (read-only) *(now)*

Strategic context: [`roadmap.md`](roadmap.md) → *v0.4*. The barrier to adoption is the analyst's time: they won't re-enter on Vidit work they've already published to X. Two self-serve channels remove that cost: the analyst **uploads their official X archive** (full history, no API) and **tags a Vidit bot** on each new geolocation tweet. Machine-produced geolocations land as **`detected`** (visible everywhere but clearly marked) and become **`geolocated`** once the owner vouches; a direct submission or a request fulfilment is born `geolocated`.

Consent is the action itself: the upload and the tag are in-product and cover only the analyst's own posts; nothing is fetched, processed, or published for anyone who hasn't acted. Phases: **A** (extraction core + state machine), **B** (the on-ramp: backfill, bot, Detections queue, value layer), **C** (go public). Phase C couldn't open until `detected` rendered marked on every surface (shipped).

**Coverage is text-only, a stated limitation.** On a 48.5k-tweet external OSINT corpus (853 analysts) the text parser recovers **~86 %** of geolocations at ~0 % false positives (decimal dominates; DMS / hemisphere are the handled long tail). The remaining **~14 %** put the coordinate only inside the image; recovering them would need vision over every backfilled media, **out of scope** (documented to the analyst; the OCR cross-check row is a *value* add, not a recovery path).

**Landed** (see [CHANGELOG](../CHANGELOG.md) → *v0.4.0*): the Phase A detection spine (acquire / stitch / detect bricks, `assemble`, the no-persist preview, `detected` marked on every surface, dev-fed by a synthetic archive via `make seed-detections`); the request + geolocation + detection merge into one `events` lifecycle (`requested` → `detected` → `geolocated` → `closed`, the promotion apparatus removed, the `/bounties` **router** removed, request / geolocation / detection now filtered views over `GET /events`); the event-model refactor (`owner_id` split from the durable `event_geolocators` credit, `event_claims` → `event_investigators`, `submit`/`reject` → `geolocate`/`close`, `location` → `event_coords` + a new `capture_source_coords`, the `proof_images` table dropped in favour of `media(role='proof')` uploaded at publish, stored IP/UA dropped); the design-system primitives audit; the **honest source contract** (`source_url` / `source_posted_at` nullable + `ck_events_source_url_status`, the `source_url_required` gate at the `geolocate` promotion, the self-source fallback removed, archive video capture, see [`ingestion.md`](../docs/ingestion.md)); the **ingest-contract test suite per typology** ([`backend/tests/ingest_contract/`](../backend/tests/ingest_contract), fixture-driven: each typology pins `resolve_thread`'s derived fields offline plus the full archive backfill against the test DB); the **archive-upload on-ramp** (`POST /events/import-archive`, tweets-only allowlist extraction + in-browser strip, the `/submit` bulk mode, a newly-confirmed account lands on it, now backed by the **durable import worker** + completion email, see [`ingestion.md`](../docs/ingestion.md#archive-import-worker), and surfaced in the panel as a per-step progress stepper: privacy filter → upload with real byte counts → queued → extraction with live detection counts, the new `ProgressSteps` primitive, see [`design.md`](../docs/design.md#components)); the **profile insights** (the `GET /users/{username}/stats` aggregation + the profile section, the import's value payoff); and the **bot** (the "Tag @ViditBot" half of the on-ramp: greenfield paid X API client with OAuth 1.0a replies, mention pipeline with the self-only parent rollup, the `bot_mentions` idempotency ledger, attribution restricted to existing accounts with an admin-linked `x_handle` (bound to the invite code at mint, copied onto the account at registration; an unknown handle ledgers `no_account` and produces nothing, the assembled-profile minting cut), linkless in-thread reply with warnings, run by `scripts/run_bot.py` as a Railway cron, see [`ingestion.md`](../docs/ingestion.md#bot-format)). **What's left**, below: the **operational go-lives** (import worker, bot cron, conflict-sync cron, Cloudflare) and the **release** gate (thread + cut). The authenticity sample-check and `note-tweet.js` long-form bodies stay descoped to *Unscheduled candidates*, promoted back if a real archive needs them; the OCR cross-check and perceptual dedup halves of the value layer moved to v0.5; the upfront text-only recovery statement in the import panel is cut (real analyst archives carry the coordinate in the tweet text, and the zero-result message already states the limitation). The synthetic archive (`make seed-detections`) stays the dev feeder.

**Decision: the Phase C legal-review gate is cut.** Anonymous read ships without a standalone legal sign-off: the consent-first design keeps the republication surface to media each analyst chose to publish under their own handle, and the full legal stack (entity, avocat, CGU, DSA machinery) is already a v0.5 hard gate before open write. Anonymous read is now **open** at the frontend proxy (content routes public, `GET /search` opened); write and account surfaces stay behind the wall, and the `ClosedBetaBanner` stays until the closed-beta framing retires (v1.0).

**Decision: no handle-ownership verification in v0.4.** X's OAuth consent screen is too broad even at the minimal `users.read` scope ("Any account you can view, including protected accounts") for a privacy-conscious OSINT audience, and X offers no lighter identity integration: no OpenID Connect, and OAuth 1.0a "Sign in with Twitter" is *worse* (timeline + follows + settings). So OAuth handle-ownership is dropped (see the closed *Continue with X* PR, kept on its branch if ever wanted). An import therefore attributes work to a `@handle` **without proving the uploader owns it**. The exposure (fabricating an archive to impersonate a handle) is high-effort and low-likelihood in closed beta, and everything lands as `detected` draft (nothing is publicly vouched without a later geolocate). **Ownership proof + a claim/dispute pipeline move to v0.5** (needed there regardless, e.g. a geolocation stolen and imported under the wrong handle). Follow-up simplification: **assembled profiles are cut**. The bot no longer mints credential-less `users` rows from a tagged handle; it attributes only to an existing account whose `x_handle` is linked, and an unknown handle just ledgers the mention (`no_account`). The nominal link rides the invite code: the admin binds the handle at mint, delivery of the code over X DM to that handle is the implicit possession proof in closed beta, and registration copies the handle onto the account (fail-soft on a race, with `PATCH /admin/users/{id}/x-handle` as the repair path and the only other write path). Verify-by-post in v0.5 becomes the self-serve link path rather than a claim flow over minted profiles.

## Refactors

Engineering-hygiene work, not gated on any version; pick these up when they unblock the next feature or when the smell starts to bite. Order is rough priority.

| Area | Item | Why / how |
|---|---|---|
| Frontend | Persist the display preferences server-side | Settings → Display now carries three browser-local prefs (`localStorage`): accent palette, light/dark theme, help-tooltip visibility. None follows a signed-in user across devices. The client plumbing is already factored ([`attributePreference.ts`](../frontend/src/lib/attributePreference.ts) for the `data-*` reflection, [`useClientPreference.ts`](../frontend/src/hooks/useClientPreference.ts) for the `useSyncExternalStore` wiring), so a server-side move folds all three into a single user-profile preferences blob (one `GET`/`PATCH`) rather than a column per toggle. Not worth a backend round-trip while they stay cosmetic. |
| Frontend | Modernise `useEffect` patterns to satisfy `react-hooks/set-state-in-effect` | Next 16's `eslint-config-next/core-web-vitals` preset adds `react-hooks/set-state-in-effect` (a new React Compiler-integrated rule) and 15 sites in the frontend trip it today; every `setX(...)` call inside a `useEffect` body, mostly auth-gating and bootstrap-data-load flows. The Next 16 upgrade PR downgrades the rule from `error` to `warn` in [`frontend/eslint.config.mjs`](../frontend/eslint.config.mjs) so the build stays green; this row is the actual cleanup. The right move per site is one of: (a) compute-at-render (drop the effect, derive directly from props/state); (b) move the `setX` into an event handler / data-loader; (c) keep the effect but factor the setState into a `useMemo` or a state-reducer. Worst sites: [`contexts/AuthContext.tsx`](../frontend/src/contexts/AuthContext.tsx), [`components/Sidebar.tsx`](../frontend/src/components/Sidebar.tsx), all the `app/(auth)/*/page.tsx` flows. Pick each site up when you touch it for another reason; a single 15-site sweep is precisely where regressions in auth effects creep in. Restore the rule to `error` once zero sites remain. Toolchain baseline these effects type-check against: **TypeScript 6**, `tsconfig` `target: ES2017` (off the deprecated `es5` default). |
| Backend | Declare migration-only indexes on the models | Roughly half the production indexes (GIST, FTS, partials) exist only in migration files; `alembic revision --autogenerate` emits `DROP INDEX` for each, so a contributor trusting the autogenerate diff strips them from prod.
